# 报告结构蓝图

本参考定义 `multi-source-analysis-workbench` 在完成标准化与分析后，面向“结论/建议/报告”交付场景的默认报告契约。

除非用户明确只要单一格式，默认报告包应包含：
- 一份本地 `HTML` 报告
- 一份本地 `Excel` 工作簿

两种格式应由同一组逻辑表驱动生成。

## 1. 目标

报告应是“分析交付物”，而不是原始导出，也不是营销落地页。

报告应帮助读者回答：
- 使用了哪些数据
- 采用了什么分析方法
- 最重要的模式和结论是什么
- 可执行的行动或机会是什么
- 对应证据在哪里

## 2. HTML 结构

HTML 页面应采用白底分析风格，分区清晰、配色克制、信息密度高。

必备模块：
1. 顶部摘要
2. 数据范围与样本说明
3. 方法说明与编码框架
4. 按维度的核心发现
5. 分段或旅程视图
6. 建议与机会待办
7. 证据附录

### 2.1 顶部摘要

目标：
- 快速建立阅读上下文
- 明确分析范围
- 突出最高优先级结论

最小内容：
- 报告标题
- 样本说明
- 来源总数或记录总数
- 3-5 条关键发现
- 顶级优先事项或主要风险

### 2.2 数据范围与样本说明

目标：
- 解释“实际纳入了什么数据”
- 披露关键排除项与局限

最小内容：
- 来源类型分布
- 纳入与排除说明
- 样本 caveats（如部分覆盖、低星样本、单渠道样本、OCR 阻塞等）

### 2.3 方法说明与编码框架

目标：
- 展示原始文本如何变成结构化结论

最小内容：
- 分析单位
- 主要维度/分类框架
- 标签定义或判定说明
- 证据等级说明

### 2.4 按维度的核心发现

目标：
- 呈现主题/问题/分类桶的主汇总视图

最小内容：
- 排序后的发现卡片或表格
- 数量与占比
- 代表性证据
- 解释与影响说明

若存在多个维度，可设置主维度 + 支撑维度分区展示。

### 2.5 分段或旅程视图

目标：
- 显示模式在阶段、分群、产品或渠道上的聚集位置

最小内容：
- 一张阶段或分段汇总表
- 一张图表或矩阵
- 聚集点与转折点说明

若任务无分段/旅程视角，可替换为 `source comparison` 或 `document section map` 模块。

### 2.6 建议与机会待办

目标：
- 将证据转化为行动

最小内容：
- 优先级
- 建议项
- 建议依据
- 预期影响
- 证据链接

### 2.7 证据附录

目标：
- 保障报告可审计

最小内容：
- 来源文件
- 记录 ID
- 标题
- 原文或摘录
- 标准化/翻译/解释文本（如适用）
- 应用标签

建议支持筛选：
- 关键词
- 来源类型
- 主分析维度
- 分段或旅程阶段

## 3. Excel 工作簿结构

工作簿应与 HTML 逻辑一致，不应退化为原始数据倾倒。

建议工作表顺序：
1. `报告总览`
2. `数据范围`
3. `方法说明`
4. `核心发现`
5. `分段或旅程`
6. `行动建议`
7. `证据附录`

若领域有专属命名，可调整 sheet 名称，但应保持同等语义用途。

最小工作簿规则：
- 白底、克制配色
- 各数据表冻结表头
- 长文本换行显示
- 控制列宽，避免无边横向滚动
- 行高随换行内容自适应增长
- 需要图表时使用原生 Excel 图表
- 表格在默认缩放下可读

## 4. 图表要求

图表应服务决策，建议包含：
- 来源结构图
- 发现分布图
- 分段/旅程集中度图
- 严重度或优先级图
- 方法论需要时的矩阵或热力图

不要放纯装饰性图表。

## 5. 命名规则

推荐文件名：
- `report-YYYYMMDD-HHMMSS.html`
- `report-YYYYMMDD-HHMMSS.xlsx`

若用户未指定保存位置：
- 保存到工作区 `output/`
- 同一次生成的 HTML 与 Excel 使用同一时间戳

## 6. 风格规则

- 默认语言遵循用户工作语言；当前环境可默认中文
- 页面应呈现“分析报告感”，而非产品营销页
- 优先保证清晰、可比、可追溯，而非装饰效果
- 附录应保留足够证据以支撑主结论复核

## 7. 完成检查

交付前确认报告能清楚回答：
- 分析了哪些来源
- 使用了什么方法
- 核心结论是什么
- 结论集中在哪里
- 下一步建议做什么
- 证据在哪里可查
